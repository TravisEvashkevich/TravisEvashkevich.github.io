Vehicle Pipeline for content creation

Accomplishments

Creating 3d vehicles as closely as possible to their real life counterparts Players should be able to scan a Hot Wheels vehicle and see the exact same thing in game ready to play with.

    Pipeline for all vehicles in game
  • Standalone Python
    • Scripts for folder structure/naming convention based on incoming vehicles from Mattel. This reduces human error when a brand new vehicle is to be made.
  • Zbrush
    • Zscript that handled settings/button clicks etc. to produce a more "bake ready" high poly mesh. This got artists 90% of the way there from the low poly mesh which in most cases was not suitable for baking purposes (even if subdivided)

    • Importing template meshes, exporting final vehicle for Unity (imports have 0 rotation values on import)

    • Various utility scripts
  • Substance Painter
    • Setup Templates for Painter scenes
    • Documentation on workflow for applying Decals to vehicle for each variation of the vehicle (same mesh, different texture set)
  • Shader work (Unity):
    • Wheel Shader that allowed for input of a channel packed texture to define different colour masks + text, Wheel shadowing (for shadows due to wheelwell and ground shadowing)
    • Hologram Shader that allows for customization of hologram images in game with channel packed textures that allowed for editor control of timing for fades and otheWr effects
  • Unity Scripting
    • Import script that handled 95% of work related to importing and setting up a vehicle in engine from created assets. This would trawl our source art folder, find any missing models or textures and import, setup folder structures, create and apply materials with correct naming convention with no needed input from a user
    • Scripts to help aid in work such as Quick Load/Unload of wheel assets on cars

About

I am a Technical Artist in the Games Industry that supports teams of artists to enable them to do work without worrying about the monotonous busy work of their job. I aim to lessen the amount of clicks and "monkey work" that is involved in day to day work. This generally means making tools/scripts that will handle as much work as possible in as few clicks as possible.

Creating tools that can feed data into each other to achieve the desired end result is my day to day. Aiming to make tools as reusable across projects and Digital Content Creation packages is a goal that I chase.
